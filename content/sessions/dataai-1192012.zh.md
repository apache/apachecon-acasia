Most Retrieval-Augmented Generation systems look impressive in demos and then quietly degrade in production. The problem is usually not the LLM. It is the data plane underneath it. Documents change, chunking strategies evolve, embeddings get regenerated inconsistently, retrieval quality drifts across domains, and teams often cannot reproduce why one version of the system answered well while another regressed.

This session shows how to build a reproducible and debuggable RAG pipeline using Apache projects. I will walk through an end-to-end architecture where Apache Kafka captures live query and feedback streams, Apache Iceberg versions corpora, chunks, embeddings, and evaluation datasets with full time-travel support, Apache Spark performs large-scale chunking, enrichment, and offline retrieval evaluation, and Apache Solr provides hybrid retrieval across lexical and vector search. The focus is not on high-level diagrams. It is on the engineering details that determine whether a RAG system can actually be trusted.

We will go deep into issues that are often skipped in AI talks but matter enormously in practice: schema design for chunk and embedding lineage, handling re-embedding without losing comparability, building evaluation datasets that do not accidentally leak, replaying historical queries against older retrieval states, diagnosing recall regressions by slice rather than relying on a single average score, combining BM25 and vector retrieval without masking failure modes, and using Iceberg snapshots to answer the most important production question of all: what exactly changed?

The session includes a realistic failure scenario where a retrieval pipeline appears to improve overall metrics while becoming materially worse for a high-value query segment. I will show how to detect that regression, trace it back to a data change, and compare alternate retrieval configurations in a way that is reproducible and explainable. Attendees will leave with concrete design patterns for building AI systems whose behavior can be audited, benchmarked, replayed, and improved using open source Apache technologies rather than opaque vendor tooling.
